Backend API Change Request — User Role Field
=============================================
Date: 2026-05-23
From: Frontend Team
Priority: Required for 3-layer architecture implementation

PROBLEM
-------
The frontend is implementing a 3-layer routing architecture (Superadmin / Senior / User).
Each user MUST be assigned to exactly one layer, and the frontend needs to know which
layer a user belongs to at login time to route them to the correct URL segment:

  /superadmin/*  →  System-level admin (God Mode, all tenants)
  /senior/*      →  Tenant admin (manage their own tenant's users/roles)
  /user/*        →  Standard operator (day-to-day wallet operations, RBAC-restricted)

Currently the only way to identify a user's layer is:
  - Superadmin: tenant_id === "1" (hardcoded convention)
  - Senior: No way to detect (frontend guesses by convention)
  - User: Default fallback (anyone who isn't superadmin)

This is fragile and cannot reliably distinguish Senior from User.

   The User entity needs a new field:

     role: string   // One of: "superadmin", "senior", "user"

   This should be stored in the database and set when creating a user.
   Default value for new users: "user"

ROLE VALUES & BUSINESS RULES
-----------------------------

  "superadmin"  →  System-level admin. tenant_id MUST be "1".
                   Only superadmins can create other superadmins.
                   Can access data across ALL tenants.

  "senior"      →  Tenant admin. Can manage users, roles, and permissions
                   WITHIN their own tenant_id only.
                   Cannot access /superadmin routes.
                   Can view all financial data for their tenant.

  "user"        →  Standard operator. Access restricted by RBAC actions
                   assigned to their TenantRole.
                   Cannot access /senior or /superadmin routes.

VALIDATION RULES (suggested):
  - A user with role "superadmin" MUST have tenant_id = "1"
  - A user with role "senior" or "user" MUST have tenant_id != "1"
  - Only a superadmin can change another user's role
  - A senior can create "user" and "senior" users within their own tenant
  - A senior CANNOT create "superadmin" users
  - Changing a user's role from "user" to "senior" (or vice versa) should
    be allowed only by superadmin or a senior of the same tenant


FRONTEND IMPACT
---------------
Once the backend returns `role`, the frontend will:

  1. Store role in the auth cookie (alongside access_token)
  2. Read role in Next.js middleware to enforce route guards:
     - role === "superadmin" → only /superadmin/* accessible
     - role === "senior" → only /senior/* accessible
     - role === "user" → only /user/* accessible
     - No valid role → redirect to /login
  3. Show layer-appropriate sidebar navigation
  4. Replace the hardcoded isSuperAdmin() = (tenant_id === "1") check
  5. Use role for UI rendering decisions (hide/show admin features)

The frontend CANNOT proceed with the 3-layer routing implementation
until the backend returns the role field in /login and /me responses.
